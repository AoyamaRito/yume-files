AI foundation file format

.yume.js keeps context small, traceable, and useful.

YUME means 夢 — a creator's dream, made tangible.

yume-files is a zero-dependency reference implementation for AI-facing files. It puts current content, history, refs, notes, impact analysis, and runtime pinning into one portable JavaScript file.

View on GitHub Try the commands

Why

AI output improves when the context boundary is clear.

Large context is not automatically good context. AI work often mixes stale notes, old drafts, unrelated specs, and repository-wide history. `.yume.js` aims to hand the model only the Block it needs, plus the refs and reasons that matter.

1

One file as a durable unit

Current content, append-only versions, mutable notes, and runtime selection live together.

↔

Refs define the reading path

Import refs, explicit `// @ref:` lines, tags, and graph checks make nearby context visible.

↓

Less token waste

Use `heavy` and `impact` to avoid sending unrelated files and noisy history to the model.

Format

A `.yume.js` file is normal JavaScript with an embedded history substrate.

The file remains readable with a JS parser. The runtime does not need a database, server, AST package, or sidecar store.

minimal shapezero dependency
// @yume-format: 1
export const __block = {
  "id": "chapterOne",
  "type": "draft",
  "runtime": { "name": "yume", "version": "001" },
  "versions": [
    {
      "content": "...",
      "refs": [{ "kind": "uses", "target": "./world.yume.js" }],
      "tags": ["chapter", "draft"]
    }
  ]
};

// === HEAD ===
// @ref: uses ./world.yume.js
export const chapterOne = "...";
// === /HEAD ===

Workflow

Expand only what matters. Apply only what changed.

`heavy` creates an AI-editable view from a root Block and its dependencies. `heavy-apply` writes the edited view back and appends new versions only where content changed.

runtime pinning

Each file declares the runtime version it expects, so older files can stay readable.

Fiction Writing

A long novel can become a graph of small, stable creative files.

World, characters, plot rules, style, and chapters can each be a Block. The model reads the few Blocks that matter instead of the whole project.

novel.world.yume.js World rules, history, geography, constraints.
mira.character.yume.js Voice, motives, secrets, relationship state.
style.guide.yume.js Sentence rhythm, forbidden phrases, tone.
chapter-07.draft.yume.js Draft text with explicit refs to world, character, and style.

Change a rule, then ask what moves.

impact novelWorld 2 . shows the chapters, style files, or workflows that reference the world Block.

refs-check . catches broken links before the AI writes from stale assumptions.

Principles

Stable substrate, evolving runtime.

Keep the physical format boring.

It is a JavaScript file with a clear header, canonical JSON Block data, HEAD, and BOOT regions.

Keep history append-only.

Content versions are immutable. Explanations live in mutable notes so intent can be corrected later.

Keep the runtime local and zero-dep.

The v001 scanner avoids AST package dependencies and favors portable folder movement.

Keep AI context narrow.

Use refs, heavy, impact, and refs-check to send the model the smallest useful slice.

Commands

Try the reference runtime.

Node.js 20 or newer is enough. The runtime ships in this repository.

Run the full e2e suite

npm test

Show history and current content

node examples/hello.fn.yume.js history
node examples/hello.fn.yume.js show head

Expand context for AI editing

node examples/hello.fn.yume.js heavy hello 1 .

Apply an edited heavy view

node examples/hello.fn.yume.js heavy-apply view.txt hello 1 . \
  --note "AI edited view"

Check reverse impact

node examples/hello.fn.yume.js impact hello 1 .

Validate the refs graph

node examples/hello.fn.yume.js refs-check .
node examples/hello.fn.yume.js refs-check . --json
