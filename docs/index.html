AI foundation file format

A JavaScript file that owns its own version history — for AI to read, edit, and write back.

YUME means 夢 — a creator's dream, made tangible.

Each `.yume.js` file is one Block: the current state lives in a HEAD region, every previous version sits in an append-only `versions[]` array, and the runtime is pinned right inside the file. AI agents call its commands; you keep talking in natural language.

View on GitHub Try the commands

Why

When AI edits a file, the "why" disappears.

Every AI edit overwrites the last draft. The reason, the prior intent, the abandoned attempt — gone unless someone remembers to commit. `.yume.js` keeps that history inside the file itself, so no separate commit step is required.

1

One file, full history

Each `.yume.js` holds one Block. Every version lives inside the same file, append-only.

↔

AI-callable, in place

The file is its own CLI. Agents commit, roll back, and edit through natural sub-commands; the file rewrites itself.

↓

Portable, no service

Move it, copy it, share it. No database, no server, no installer — just Node and the file.

Usage

A simple way to use yume-files with an AI agent.

Start by giving the agent the repository and one clear instruction. The runbook tells it what to read, how to edit, and how to verify the result.

Ask it to read the runbook first.

Point the AI at AGENTS.md and runAndReadMe.aiDoc.yume.js. It should consult BLOCKFILE.aiDoc.yume.js only for exact format details.

Describe the file you want.

Use domain-shaped names such as novel.world.yume.js, review.workflow.yume.js, doc.template.yume.js, or chapter.draft.yume.js.

Require validation before it stops.

For yume files, the AI should edit HEAD, commit through the runtime, then run validate, refs-check ., and npm test when needed.

What it fixes

The familiar AI-development headaches, addressed in the format itself.

These are the workflow problems `.yume.js` was designed to remove. Each pain has a specific structural answer in the file format — not a tool around it, but the format itself.

Pain

AI edits silently overwrite previous drafts. The earlier version is gone.

→
Fix

Each Block keeps an append-only versions[] inside the file. No edit is lost.

Pain

Sending the whole codebase burns tokens — most of it is irrelevant to the task.

→
Fix

Pain

The reason for a change disappears the moment the edit happens. Commit messages rot or get skipped.

→
Fix

notes are mutable and can be attached to a version after the fact, without changing its hash.

Pain

One AI task touches many files. Later it is hard to tell which changes belong together.

→
Fix

Each AI operation gets a shared applyId. apply-show reproduces the full change set across files.

Pain

AI hallucinates functions or files that do not exist, or that have moved.

→
Fix

refs-check validates the graph: dangling paths, duplicate ids, cycles, broken Blocks.

Pain

A breaking edit is hard to undo across many files.

→
Fix

rollback returns any Block to a prior version. History lives in the file, not in git.

Pain

Granting AI full git access is risky — but without it, the model cannot see the file's past at all.

→
Fix

History sits inside each .yume.js. Without any git permission, the AI reads every past version — mistakes included — and corrects the Block from full historical context.

Core mechanism

One Block per file. Each Block carries its own version history inside it.

.yume.js is a per-Block version control system that lives inside the JavaScript file itself. The Block's `versions[]` array is append-only; the HEAD region exports the current state. Multiple files form a graph of independently-versioned Blocks — no shared timeline, no central history server.

chapter-07.draft.yume.js

block.id = chapterSeven
versions[0] initial draft
versions[1] tightened the opening
versions[2] added ref to mira.character
versions[3] added literary-minimal style ref HEAD

No git, no database. Each file is its own micro-repository. Move it, share it, revert it — the full history travels with the content.

Format

A `.yume.js` file is normal JavaScript with an embedded history substrate.

The file remains readable with a JS parser. The runtime does not need a database, server, AST package, or sidecar store.

minimal shapezero dependency
// @yume-format: 1
export const __block = {
  "id": "chapterOne",
  "type": "draft",
  "runtime": { "name": "yume", "version": "001" },
  "versions": [
    {
      "content": "...",
      "refs": [{ "kind": "uses", "target": "./world.yume.js" }],
      "tags": ["chapter", "draft"]
    }
  ]
};

// === HEAD ===
// @ref: uses ./world.yume.js
export const chapterOne = "...";
// === /HEAD ===

Architecture

Each file boots itself, then hands control to its pinned runtime.

A `filename.domain.yume.js` file declares the runtime version it expects in its embedded Block. Running it with Node fires a small BOOT region that dynamically imports the matching `runtimes/verNNN.handle.yume.js` and hands the runtime the file's URL, the Block data, and argv. The runtime then reads, parses, and rewrites the same file — appending versions, updating HEAD, editing notes — without any external service.

BOOT regionruntime delegation
// @yume-format: 1
export const __block = {
  "id": "chapterSeven",
  "runtime": { "name": "yume", "version": "001" },
  "versions": [ /* ... */ ]
};

// === HEAD ===
export const chapterSeven = "...";
// === /HEAD ===

// === BOOT ===
if (import.meta.url === `file://${process.argv[1]}`) {
  const path = __block.runtime.path
    ?? `../runtimes/ver${__block.runtime.version}.handle.yume.js`;
  const rt = await import(path);
  await rt.cli(import.meta.url, __block, process.argv);
}
// === /BOOT ===
01

The Block carries its runtime version

__block.runtime = { name: "yume", version: "001" } pins the version. Older files keep their old runtime, no matter what newer files do.

02

BOOT loads the matching runtime

node chapter-07.draft.yume.js commit dynamically imports runtimes/ver001.handle.yume.js. No installer, no global service — just Node and local files.

03

The runtime self-analyzes and rewrites the file

It opens the same file, parses its Block + HEAD + BOOT regions, mutates them in place (append a version, update HEAD, edit notes), and writes the file back. The file is its own micro-repository.

Workflow

Expand only what matters. Apply only what changed.

`heavy` creates an AI-editable view from a root Block and its dependencies. `heavy-apply` writes the edited view back and appends new versions only where content changed.

Principles

Stable substrate, evolving runtime.

Keep the physical format boring.

It is a JavaScript file with a clear header, canonical JSON Block data, HEAD, and BOOT regions.

Keep history append-only.

Content versions are immutable. Explanations live in mutable notes so intent can be corrected later.

Keep the runtime local and zero-dep.

The v001 scanner avoids AST package dependencies and favors portable folder movement.

Keep AI context narrow.

Use refs, heavy, impact, and refs-check to send the model the smallest useful slice.

Commands

The AI calls these. You stay in natural language.

These commands are the interface an AI assistant — Gemini CLI, Claude Code, Codex, and so on — uses while working on your `.yume.js` files. As the human, you keep talking in natural language; the agent handles the snapshot, rollback, and heavy/heavy-apply loop behind the scenes. The `.yume.js` file is its own CLI, so there is no separate tool to install. Node.js 20 or newer is enough.

Read the AI runbook

npm run runbook
node runAndReadMe.aiDoc.yume.js validate

Snapshot the current HEAD as a new version

node chapter-07.draft.yume.js commit \
  --note "tightened opening"

Browse history, show a version, diff two versions

node chapter-07.draft.yume.js history
node chapter-07.draft.yume.js show head
node chapter-07.draft.yume.js diff 0 -1

Roll back to a previous version

node chapter-07.draft.yume.js rollback -1 \
  --note "restore earlier draft"

Add or correct an intent note on a version

node chapter-07.draft.yume.js note-add head \
  "why this version exists"

Expand context for AI editing (heavy)

node chapter-07.draft.yume.js heavy chapterSeven 1 .

Apply the AI-edited view back (heavy-apply)

node chapter-07.draft.yume.js heavy-apply view.txt \
  chapterSeven 1 . --note "AI revised opening"
